Phase 3 Server Islands Pilot Report
===================================

- Component Selected: AI/chat widget (`PersistentAIBottomBar.astro` / `AIAssistantButton.astro`)
- Why it was selected: It provides dynamic interactive functionality that shouldn't block the main page's initial paint. SEO is unaffected because chat content does not need to be indexed.
- Was `server:defer` used?: Deferred (Not yet used in code). Astro's Server Islands (`server:defer`) require a server adapter and `output: 'server'` or `output: 'hybrid'`. Since the current architecture is strictly static, Server Islands cannot be built right now without breaking the static paradigm.
- Fallback Behavior: If implemented, the widget would render a static skeleton or loading button as the fallback while the Server Island streams in the dynamic AI content.
- Performance Result: A proper Server Island ensures the main static HTML returns instantly from the CDN edge, while the dynamic component loads concurrently in the background.
- Expand Later?: Yes, if the project is updated to Astro's hybrid rendering mode, Server Islands will be a great fit for pricing recommenders, audit forms, or authenticated UI.

Acceptance Check:
- Page remains fast/static: Documented as safe.
- Dynamic widget loads separately: Documented as the intention for hybrid.
- No SEO content is hidden inside island: AI Chat is not SEO content.
- No Sanity quota leak: Verified (chat widget uses an internal setup, not Sanity content queries for chat messages).
