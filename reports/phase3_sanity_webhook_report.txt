Phase 3 Sanity Webhook Report
=============================

- Webhook Filter: Blog posts, case studies, services, locations, and global settings.
- Webhook Projection:
  ```json
  {
    "_type": _type,
    "_id": _id,
    "slug": slug.current,
    "updatedAt": _updatedAt
  }
  ```
- Endpoint Path: `/api/revalidate` (deferred)
- Secret Header Name: `x-sanity-webhook-secret` (deferred)
- Cache Tags to Invalidate: `post:${slug}`, `blog:index`, `service:${slug}`, `sitemap`, `homepage`
- Fallback Deploy-Hook Strategy: Because the Astro output is fully static (`output: 'static'`) and lacks an active API directory, an API route for `api/revalidate` cannot be safely exposed without converting the project to `output: 'hybrid'` or `output: 'server'`. For now, we will maintain the fallback strategy: a Sanity Deploy Hook triggers a full Vercel build whenever production content is published.
- Manual Sanity Dashboard Setup Steps:
  1. In Sanity project settings, navigate to API > Webhooks.
  2. Create a webhook for Vercel Deploy Hook (the current working method).
  3. When the architecture upgrades to `hybrid`, create a new webhook pointing to `https://theeduassist.com/api/revalidate`.
  4. Use the projection defined above.
  5. Add a custom header: `x-sanity-webhook-secret` with a securely generated token.

Acceptance Check:
- Webhook payload is small: Documented.
- Endpoint is protected: Documented (via secret).
- No secrets in repo: Verified (deferred implementation means no secrets added).
- No browser access to secret logic: Verified.
- No broad full-site invalidation unless necessary: Fallback deploy hook does full invalidation, which is safe and necessary until tag invalidation is built in a hybrid environment.
