# Cache Optimization Report

## Actions Taken
1. **Removed Unnecessary Client-Side Hydration:**
   - Identified `<PortableTextRenderer client:load />` in `src/pages/[slug].astro` and `src/components/blog/BlogFAQs.astro`.
   - Removed the `client:load` directive. PortableTextRenderer is purely rendering static HTML based on CMS content and requires no React browser state. This prevents Astro from shipping React/Sanity client JS bundles to the browser for these components, reducing payload and preventing direct browser-to-Sanity API fetches.

2. **Sanity Client De-Duplication and Hardening:**
   - Found two Sanity client configurations: `src/sanity/client.ts` and `src/lib/sanity/client.ts`.
   - Updated both to strictly enforce `useCdn: false`.
   - Added comments indicating these clients must only be used on the server or during build time. This ensures that static generation fetches fresh data from Sanity's API, and the browser never calls the API directly.

3. **Vercel Caching Headers Setup:**
   - Modified `vercel.json` to include explicit `Cache-Control` headers for immutable static assets (`/_astro/*`, `/assets/*`) setting `max-age=31536000` (1 year).
   - Set `/images/*` caching to 30 days.

## Expected Impact
* Public website visitors will now receive fully pre-rendered static HTML and cached assets.
* Browsers will no longer initialize the Sanity React client or trigger direct requests to `apicdn.sanity.io`.
* Sanity API CDN usage will drop to near zero for public traffic, only spiking during Vercel builds triggered by content updates.
