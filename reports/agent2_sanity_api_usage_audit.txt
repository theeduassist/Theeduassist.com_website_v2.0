Sanity API Usage & Browser Network Safety Audit
===============================================

Audit Results
-------------
- Checked the `src/sanity/client.ts` wrapper. It defines `sanityBuildClient` (useCdn: false) and `sanityCdnClient` (useCdn: true) properly.
- The `fetchFromSanity` helper allows options to dictate which client to use, failing gracefully if variables are missing.
- Queried the codebase for `client:load` and `client:visible`. None of them import Sanity client inside browser components. The usage is strictly for Firebase, Turnstile, and other UI logic.
- Analyzed `src/components/blog/AllArticlesGrid.astro` and `src/pages/data/blog/page-[page].json.ts`. They use statically generated JSON files (`/data/blog/page-*.json`) instead of querying `api.sanity.io` or `apicdn.sanity.io` from the browser. This prevents token exposure and avoids API rate limits.
- Optimized images through Sanity use `cdn.sanity.io/images/...` which is the expected standard for displaying Sanity-hosted images and safe for public exposure.

Verdict: PASS. No Sanity API query logic or tokens are exposed to the browser bundle.
