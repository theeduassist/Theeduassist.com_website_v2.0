Phase 3 Monitoring & Alerting Report
======================================

### Daily Checks
- **Sanity Request Increase:** Monitor Sanity's dashboard (API tab) for sudden spikes in requests or CDN usage. Spikes might indicate a missing `Cache-Control` header or rogue client-side fetching.
- **Vercel Error Rate:** Check Vercel logs for 500 errors or failed builds to ensure static rendering processes aren't timing out.
- **Sitemap Status:** Ensure `https://theeduassist.com/sitemap.xml` returns 200 and correctly lists live URLs without including drafts.
- **Robots Status:** Ensure `https://theeduassist.com/robots.txt` returns 200 and points to the sitemap.
- **Lead Form Working:** Submit a test through the primary lead form (or audit form) to ensure form submission handlers (e.g., Netlify Forms, Formspree, or custom API) are responding correctly.

### Weekly Checks
- **Search Console Indexing:** Review Google Search Console for new 404s, coverage errors, or leaked preview subdomains being indexed.
- **Top Pages:** Review analytics (e.g., Google Analytics or Vercel Web Analytics) to ensure top landing pages (Home, Services, Locations) receive traffic and haven't dropped.
- **404s:** Review Vercel analytics or logs for frequent 404 paths. Implement redirects in `vercel.json` for any common broken links.
- **Blog Performance:** Ensure pagination is fast, static JSON loads properly, and images display correctly.
- **Sanity Quota Trend:** Ensure API usage remains well within monthly limits (using `useCdn: false` for builds and static pages ensures low usage).

### After-Deploy Checks
- **Sanity 24-48 Hour Quota Check:** Check Sanity usage 24 and 48 hours after a large deployment or traffic event.
- **Google Search Console Validation Check:** Run the URL Inspection tool on the homepage and a blog post to verify proper rendering and indexing permissions.
- **Vercel Dashboard Check:** Ensure the deployment completed successfully, cache tags (if implemented) were recognized, and Edge Requests look normal.
- **Security Check:** Run `npm run validate:env` locally or in CI to ensure no secrets (`[SANITY_TOKEN]`) leaked into the codebase.
