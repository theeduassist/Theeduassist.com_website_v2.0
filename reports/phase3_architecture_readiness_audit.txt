Phase 3 Architecture Readiness Audit
====================================

Architecture Overview:
- Astro Output Mode: Fully static. The project is currently configured to build as static HTML (`output: 'static'` implied or explicitly set).
- Vercel Adapter Configuration: Uses `@astrojs/vercel` configured as a standard adapter. Since `output: 'server'` is not enabled, Vercel acts as a static host for the pages.
- Static vs Server/Hybrid Routes: All current routes are static. Pages use `export const prerender = true;` or rely on the global static output mode.
- Existing API Routes: None. The `src/pages/api` directory does not exist.
- Current Sanity Clients: Configured in `astro.config.mjs` and custom lib files with `useCdn: false` to ensure build-time fetching does not incur unnecessary CDN costs, and drafts/hidden statuses are filtered out.
- Current Preview/Draft Behavior: There is currently no active preview route for editors. Drafts are safely filtered out at build time, preventing them from being generated.
- Current Webhook Setup: No custom API webhook endpoints exist in the codebase. The project currently relies on Vercel's GitHub integration or Sanity-to-Vercel deploy hooks (full rebuilds).
- Current Vercel Deploy Hooks: Expected to be standard full-site rebuild triggers from Sanity.
- Current Cache Headers: Configured via `vercel.json` for assets, images, and JSON payloads (e.g., `Cache-Control: public, max-age=3600` for blog JSON).
- Current Sitemap/Robots Behavior: Fully static. `sitemap.xml` is generated by `scripts/generate-sitemap.mjs` during the `prebuild` step. `robots.txt` is served statically.
- Current Blog Pagination/Static JSON Setup: Fully static JSON generation for "Load More" to prevent browser-side Sanity calls and reduce payload.

Readiness Report:
- What is fully static: The entire website (all pages, blog index, pagination, JSON payloads, assets).
- What is CMS-driven: Blog posts, case studies, and potentially specific services.
- What could use tag-based invalidation later: Blog detail routes (`/blog/[slug]/`), blog index (`/blog/`), specific CMS-driven service routes (e.g., `/services/ai-ethics-corporate-training/`), and the sitemap.
- What should stay static forever: Legal pages, `robots.txt`, core static assets, and purely structural pages without CMS data.
- What should never be cached: Any future preview routes (`/preview/*`), draft API endpoints, and sensitive editor views.
- What should never use Server Islands: SEO content, main page copy, blog article bodies, blog cards, schemas, location page content, and the site footer.

Conclusion:
The project is structurally sound but currently lacks the `output: 'hybrid'` or `output: 'server'` Astro configuration required to natively handle advanced server-side cache invalidation or webhooks in code. For this phase, cache tags can only be applied via static Vercel headers (`vercel.json`). Advanced webhooks and Server Islands will require enabling hybrid mode or are deferred to a later architectural shift.
